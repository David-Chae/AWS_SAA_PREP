# ⭐ Amazon Aurora를 쉽게 이해하기

## 1. **Aurora는 RDS의 ‘업그레이드 버전’ 같은 것**

Aurora는 PostgreSQL·MySQL과 호환되는 데이터베이스인데,
일반 RDS보다 **더 빠르고 자동화된 구조**를 갖고 있어.

핵심 특징은 **저장(Storage)과 컴퓨팅(DB 인스턴스)을 완전히 분리했다는 것**.
이게 성능, 안정성, 확장성을 모두 크게 끌어올려.

---

# 🌩 2. Storage 구조 — ‘6중 백업을 가진 튼튼한 금고’

Aurora는 데이터를 **3개의 Availability Zone(AZ)** 에
**각각 2개씩, 총 6번 복제해 둬.**

→ 즉,

* 한 곳이 고장 나도 문제 없음
* 저장소는 항상 스스로 오류 복구(Self-healing)
* 저장 공간은 자동으로 늘어남 (Auto-scaling)

이 구조는 수정할 수 없을 정도로 **고정적으로 안정성을 극대화**해 둔 아키텍처야.

---

# ⚙ 3. Compute 구조 — ‘여러 대의 서버가 하나의 팀처럼 움직임’

DB 인스턴스(컴퓨팅 서버)는 **클러스터 형태**로 묶여 있어.

* 원하면 여러 AZ에 걸쳐 인스턴스를 둘 수 있음
* 읽기 전용 인스턴스(Read Replicas)는 자동으로 늘릴 수 있음 → 읽기 부하 대처 쉬움

즉, **읽기 트래픽이 몰릴 때 자동으로 서버가 늘어나서 감당하는 구조**.

---

# 🎯 4. 엔드포인트 — ‘역할별 진입로가 따로 있음’

Aurora는 클러스터 기반이라 각 역할에 맞는 전용 엔드포인트를 제공해.

* **Writer Endpoint**: 쓰기(INSERT/UPDATE/DELETE) 전용
* **Reader Endpoint**: 읽기(SELECT) 전용, 여러 리드 인스턴스에 자동 분산

따라서 애플리케이션 입장에서는 복잡한 로드밸런싱을 신경 쓸 필요 없어.

※ 보안·모니터링·백업 방식은 RDS와 거의 동일해.

---

# 📦 5. Aurora의 추가 기능들

## ✔ (1) Aurora Serverless

* 사용량이 들쭉날쭉할 때 매우 유리
* 서버를 직접 고르는 과정 없이 **필요할 때 자동으로 용량이 늘었다 줄었다**
* 완전히 관리형 → 비용도 효율적

→ 예: “밤엔 유저 거의 없음, 낮에는 폭주” 이런 앱

---

## ✔ (2) Aurora Global Database

* 최대 16개의 읽기 인스턴스를 지역별로 배치 가능
* 지역 간 데이터 복제가 **1초 미만**
* 한 지역이 죽으면 다른 지역을 바로 주요 리전으로 승격

→ 글로벌 서비스(전 세계 유저)에서 엄청난 이점.

---

## ✔ (3) Aurora Machine Learning

Aurora에서 직접 SageMaker/Comprehend 모델을 호출할 수 있음.
→ DB 데이터 기반의 AI 기능을 자연스럽게 연동 가능.

---

## ✔ (4) Aurora Database Cloning

테스트용 DB를 만들 때, 스냅샷 → 복원 방식보다 훨씬 빠름.
→ 기존 DB를 기반으로 바로 “복사(copy-on-write)”해서 새 클러스터 생성.

→ 개발·스테이징 환경 구축이 매우 쉬움.

---

# 🧰 6. 언제 쓰는 서비스인가?

Aurora는 RDS와 거의 같은 목적(OLTP, SQL 기반 서비스)에 쓰지만:

* 더 높은 성능
* 자동화된 확장성
* 글로벌 복제
* 고급 기능들

덕분에 RDS보다 **“엔터프라이즈급 환경”** 에 매우 적합해.

---
